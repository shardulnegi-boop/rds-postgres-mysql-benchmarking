What Happened to MySQL

Round 1 (4 workers): 66 MB/s — InnoDB handling LOAD DATA, redo log keeping up
Round 2 (8 workers): 51 MB/s — Redo log filling faster, checkpoint pressure building
Round 3 (12 workers): 34 MB/s — Buffer pool saturated with dirty pages, checkpoints can't keep up
Round 4 (16 workers): 21 MB/s — Emergency checkpoint flushing, new writes queuing
Round 5+ (20-64 workers): 0 MB/s — Complete write stall. LOAD DATA commands accepted but never completed. InnoDB stuck in checkpoint loop. Did not recover even after table truncation between rounds.
Root Cause: InnoDB Redo Log Saturation
  • Each LOAD DATA LOCAL INFILE of a 1 GB file generates massive redo log entries
  • Multiple concurrent 1 GB loads overflow the redo log buffer
  • InnoDB triggers emergency checkpoint flush to reclaim redo log space
  • Checkpoint flushing can't keep up with incoming write volume
  • New writes queue behind the checkpoint — throughput drops to zero
  • This is the exact same failure mode as the production incident last year at 300 MB/s

What Happened to PostgreSQL

Why PostgreSQL survived:
  • UNLOGGED table bypasses WAL (Write-Ahead Log) entirely — no redo log bottleneck
  • COPY FROM STDIN writes directly to heap files with minimal overhead
  • PostgreSQL's buffer manager handles concurrent writers without global locks
  • Even at 64 concurrent COPY streams, the engine maintained >100 MB/s
  • Throughput peaked at 44 workers (192 MB/s) then gracefully degraded — never stalled

Caveats

1. UNLOGGED vs InnoDB: PostgreSQL used an UNLOGGED table (no WAL), while MySQL/InnoDB always writes redo logs in Aurora. This gives PG an advantage in raw write speed. In production with logged tables, PG throughput would be lower — but PG's architecture still handles concurrent writes better.
2. No indexes on either engine: Both tables had no primary key or secondary indexes. Production tables with indexes would see lower throughput on both engines, but MySQL would be disproportionately impacted due to InnoDB's clustered index requirement.
3. Bulk load only: This test measured pure write throughput using the fastest bulk load method for each engine. Real workloads with mixed reads/writes, transactions, and concurrent queries would behave differently.
4. Single writer architecture: Both Aurora engines are limited to a single writer instance. Neither can reach 1 GB/s on a single writer regardless of instance size.

Recommendation

For the MySQL → PostgreSQL migration decision:

PostgreSQL is clearly superior for high-throughput bulk writes: 2.9× peak throughput, graceful degradation vs. complete stall, and dramatically better concurrency handling. The data supports migrating from Aurora MySQL to Aurora PostgreSQL.

However, neither engine reaches 1 GB/s on a single writer. PostgreSQL peaked at 192 MB/s — that's a 5× gap from the 1 GB/s target. Switching engines alone does not solve the scaling problem.
